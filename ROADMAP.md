# Nano-FFI Roadmap

From the current `0.2.0` release to a stable `1.0.0`, in `0.1` increments.
Each milestone is a real, buildable, tested release with its own `CHANGELOG`
entry and git tag. The guiding constraint never changes: **only
`python_ext.zig` may import `<Python.h>`; everything else is pure Zig over a
C-ABI boundary, and the hot path stays branch-free via comptime trampolines.**

## Design principles (hold across every milestone)

1. **Comptime, not runtime.** New type or feature support is generated by
   `makeTrampoline` at compile time. No feature may add per-call branching to
   the dispatch hot path that could be resolved at comptime instead.
2. **One Python file.** The Python.h surface stays confined to `python_ext.zig`.
3. **Single source of truth for facts.** The version string, supported-type
   list, and signatures live in exactly one place each and are read everywhere
   else.
4. **Every release builds and passes tests on the reference platform**
   (Windows x64, Zig 0.15.2, CPython 3.14) before it is tagged.

## 0.3.0 — Type-system foundation

Expand the scalar marshalling beyond `i64`/`f64`/`bool`, and pay down the
day-3 debt.

- Add `i32`, `u32`, `u64`, `f32`, `u8` to `ArgType` and the pack/unpack tables.
- Range-checked narrowing on the Python -> Zig side (a Python int that does not
  fit the target width is a `ValueError`, not a silent truncation).
- Single-source the version string (`root.zig` -> `python_ext.zig`
  `version()`), fixing the `"0.1.0"` vs `0.2.0` drift.
- Remove the stale `TODO(day-3)` markers now that dispatch is wired.

## 0.4.0 — Strings and slices

Cross the boundary with variable-length data.

- New `str` (UTF-8 `[]const u8`) and `bytes` argument/return kinds.
- Python `str`/`bytes` -> Zig `[]const u8` (borrowed for the call's duration);
  Zig `[]const u8` -> Python `str`/`bytes` on return.
- `RawValue` gains a `slice` member (`ptr` + `len`) kept C-ABI safe.
- Ownership contract documented: Zig-returned slices are copied into Python
  objects before the call returns; no dangling views.

## 0.5.0 — Error propagation

Make Zig failures first-class Python exceptions instead of silent zeros.

- Trampolines may wrap functions returning `E!T`; a returned error maps to a
  Python exception with the Zig error name.
- Conversion failures (overflow, wrong type, bad UTF-8) raise precise
  `ValueError`/`TypeError` instead of returning `-1`/`0`.
- Every `Py*_As*` call is checked with `PyErr_Occurred` before use.

## 0.6.0 — Introspection and ergonomics

Make the module self-describing and pleasant from Python.

- `nano_ffi.list_functions()` and `nano_ffi.signature(name)` read straight from
  the live registry.
- Ship `nano_ffi.pyi` type stubs and a thin pure-Python convenience layer
  (`__all__`, docstrings, a `call`-wrapping helper with argument checking).
- `py.typed` marker for downstream type checkers.

## 0.7.0 — Multi-value returns

Let a Zig function hand back more than one value.

- Signatures gain a `rets: []const ArgType` form; trampolines pack a Zig tuple
  / multi-field result into a Python tuple.
- Backward compatible: a single `ret` still returns a bare scalar.

## 0.8.0 — Zero-copy buffers

Move array data without copying it.

- Buffer-protocol (`memoryview`) pass-through: a Python buffer is handed to Zig
  as `ptr` + `len` with no copy, for read or in-place write.
- `Buffer` (already in `allocator.zig`) is exposed as a returnable, Python-owned
  view with an explicit free contract.

## 0.9.0 — Packaging, CI, docs

Make it installable everywhere and documented.

- Linux x86_64 support finalized; wheels built for Windows, macOS, Linux via a
  reproducible CI matrix (build-from-source in CI, publish pre-built wheels).
- Full API reference generated from stubs + docstrings; a runnable benchmark
  harness that emits the README table.
- Green CI required before release.

## 1.0.0 — Stability

Freeze the surface and stand behind it.

- Public API frozen; semver compatibility guarantee from here on.
- Comprehensive Zig + Python test matrix across every supported type, error
  path, and platform.
- Documentation complete; migration notes for any pre-1.0 breaking change.
- Tagged `v1.0.0`.
